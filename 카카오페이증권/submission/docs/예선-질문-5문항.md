## 1. 무엇을, 누가, 어떤 상황에서 쓰나요? (최대 800자)

mts-stability-guard는 카카오페이증권류 트레이딩 서비스(MTS·해외주식)에서 반복되는 전산장애의 재발을 막기 위한 Codex 플러그인입니다. 공개된 장애 이력을 실패 패턴으로 구조화하고, 그 패턴을 모의(mock) 브로커 위에서 재현한 뒤 방어코드로 완화되는지를 릴리즈 전에 실행 가능한 형태로 검증하도록 돕습니다. 회사 내부 인프라나 고객 데이터에는 접근하지 않고, 사용자가 지정한 트레이딩 백엔드 코드 또는 모의 API를 대상으로 동작합니다. 주 사용자는 트레이딩·브로커리지 백엔드의 안정성을 책임지는 개발자, SRE, QA 엔지니어입니다. 사용 상황은 세 가지입니다. 첫째, 새 릴리즈를 배포하기 전에 미국장 개장 피크나 외부 브로커 장애 같은 상황을 미리 주입해 안정성을 점검할 때. 둘째, 실제 장애 사후분석(post-mortem) 이후 동일 장애의 재발을 막는 회귀·부하·장애주입 테스트를 설계할 때. 셋째, 미국 현지 중개사처럼 외부 체결 브로커에 의존하는 구간의 폴백·서킷브레이커·백프레셔 설계를 검토할 때입니다. 산출물은 사람이 읽는 요약과 기계가 읽는 JSON 리포트 두 형태로 나오고, 표준 라이브러리만 사용해 별도 설치 없이 어디서나 그대로 재실행됩니다. 즉 '무엇을 방어할지'를 공개 근거로 정하고 '실제로 방어되는지'를 코드로 증명하려는 팀이, 장애가 반복되는 서비스의 안정성을 선제적으로 강화할 때 쓰는 재발방지 도구입니다.

## 2. 왜 이 문제를 선택했나요? (최대 800자)

카카오페이증권의 반복 전산장애는 공개·검증 가능한 근거가 가장 탄탄한 문제이기 때문입니다. 금융감독원 전자금융사고보고 기준 MTS 전산장애는 2022년 4건에서 2023년 9건, 2024년 11건, 2025년 13건으로 매년 증가했고, 2026년 1~3월에도 5건으로 자료를 제출한 12개 증권사 중 최다를 기록해 금감원이 IT 부문 현장점검에 착수했습니다(CBS 노컷뉴스·다음 2026.04). 또 2025년 10월 한 달에만 4차례 장애가 발생해, 미국 정규장 개장 직후 앱 접속이 불가능했고 현지 중개사(드라이브웰스) 장애로 미국주식 체결이 막혔습니다(MBC·서울신문 2025.10). 감독당국의 현장점검 착수는 확신도가 가장 높은 '사실'이며, 회사 측 해석과 분리해 다룰 수 있습니다. 이 문제를 택한 이유는 세 가지입니다. 첫째, 근거가 감독당국의 행위라 확신도가 높습니다. 둘째, 시스템 안정성은 코딩 에이전트가 테스트·방어코드·재검증까지 정면으로 수행할 수 있는 과제입니다. 셋째, 회사 내부에 접근하지 않고 공개 이력과 모의 시스템만으로 '장애 재현 → 방어 → 완화 입증'을 해커톤 안에서 end-to-end로 시연할 수 있습니다. 특히 수수료 인하 경쟁이 한계에 이르러 업계 경쟁 축이 가격에서 UX·안정성으로 옮겨가는 국면에서, 잦은 전산장애는 고객 이탈·민원·감독 제재로 곧장 이어지는 치명적 약점입니다. 다만 장애 건수는 집계 기준에 따라 달라(예: 37건 vs 42건) 출처와 기준을 함께 표기했습니다.

## 3. 플러그인은 어떻게 작동하나요? (최대 800자)

플러그인은 세 단계 파이프라인으로 작동합니다. 1단계는 '무엇을 방어할지'를 정하는 분석입니다. scripts/analyze_incidents.py가 공개 보도 기반 장애 이력(data/incidents.sample.json)을 읽어 미국장 개장 피크, 외부 브로커 의존, 내부 시스템 장애, 해외 지연이라는 네 패턴으로 분류하고 연도별 추세와 패턴 분포를 집계합니다. 2단계는 '실제로 방어되는지'를 증명하는 재현 → 방어 → 재검증입니다. demo/broker_mock.py가 5xx·타임아웃·지연·수용량 포화를 주입하고, demo/trading_backend.py가 보호장치 없는 NaiveOrderService와 백프레셔 → 헬스 게이트 → 서킷브레이커 → 지수 백오프 재시도 → 폴백을 갖춘 GuardedOrderService를 제공합니다. demo/chaos_runner.py가 동일 장애의 before/after를 비교하고 회귀 테스트가 수치를 고정합니다. 3단계에서는 scripts/run_pipeline.py가 안전한 RUN_ID별로 입력 스냅샷과 state·analysis·chaos·checklist를 남기며, 성공 실행만 output/_learnings.md에 한 줄 학습을 추가합니다. SKILL.md는 사실/해석 분리, 공개·모의 데이터 한정, 전체 memory 대신 선택적 context 사용을 지시합니다. 모든 코드는 Python 3.9 이상 표준 라이브러리만 사용하고 네트워크 없이 동작하며, 시간 의존 로직은 주입한 clock·sleep으로 결정론적으로 검증됩니다.

## 4. AI를 어떻게 활용했나요? (최대 800자)

AI는 문제 정의부터 구현·검증까지 전 과정에 활용했습니다. 먼저 문제 정의 단계에서는 deep-research 다중 소스 검색·추출 파이프라인으로 공개 자료(감사보고서·감독당국 발표·복수 언론 보도)에서 인용문을 수집·구조화하고, 각 항목에 출처 유형별 확신도를 부여했습니다. 후보 선정은 플러그인 적합성·근거 강도·시연 가능성·임팩트라는 네 축의 판정표로 아홉 개 문제를 점수화해 'MTS·해외주식 전산장애'를 Top 1으로 채택했습니다. 구현 단계에서는 실패 패턴 분류 로직, 모의 브로커, 서킷브레이커·재시도·백프레셔 같은 방어 프리미티브, 그리고 before/after를 고정하는 회귀 테스트를 AI로 설계했습니다. 시간에 의존하는 서킷 리셋·백오프 로직도 테스트가 시간을 통제할 수 있도록 clock·sleep을 주입하는 구조로 함께 다듬었습니다. 이어 코드리뷰 에이전트를 로직 정확성과 단순성·컨벤션 두 방향으로 병렬로 돌려, 서킷브레이커 상태 전이·재시도 카운팅 같은 정확성과, 리포트가 사실을 과장하거나 해석을 사실처럼 표기하지 않는지 점검했습니다. 지적된 문구와 수치는 공개 출처와 실행 결과로 다시 대조해 수정했습니다. AI가 만든 결론을 그대로 채택하지 않고 테스트의 관찰값, JSON 수치, 기사 원문을 서로 대조하는 사람 검토를 최종 게이트로 두었습니다. 원본 대화·QA 기록은 로컬에만 두고 공개 Git과 플러그인 패키지에서 제외하며, 최종 제출 단계의 logs에는 최소 sanitized 증빙 marker만 추가합니다.

## 5. 어떻게 검증했나요? (최대 800자)

'장애 재현 → 방어코드 → 재테스트 통과'를 실행 가능한 산출물로 남겨 검증했습니다. 첫째, 공개 보도 기반 incidents.sample.json으로 분석 스크립트를 실행해 연도별 추세와 패턴 분포를 확인했습니다. 둘째, chaos_runner로 공개 장애 세 패턴을 모의 브로커에 주입해 무방비와 방어 경로를 비교했습니다. 외부 브로커 전면 장애에서는 무방비가 주문 실패 10건을 사용자에게 그대로 누출한 반면 방어 경로는 유실 0으로 강등했고, 일시적 타임아웃 6건은 전량 회복했으며, 개장 피크의 초과 주문 12건은 '보류'로 강등했습니다. 셋째, 현재 핵심 회귀 70개와 패키징 검사 5개를 검증했고, 최종 제출 단계에서 계약 검사 6개를 더해 공개 검증 81개로 고정합니다. 패키징 검사는 생성 산출물 배제, 표준 라이브러리 의존성, 문서 placeholder 부재도 확인합니다. 분석 JSON은 사건 5건·시나리오 4개, 카오스 JSON은 3개 시나리오와 guarded crashed=0을 확인했습니다. 세 CLI의 help, JSON 파싱, Python 3.9 실행, 네트워크나 추가 설치가 없는 동작도 검증했습니다. 실행 파일은 로컬 input/output만 사용합니다. 전용 redaction 엔진이 없으므로 공개·모의 데이터만 허용하고 계정·토큰·개인정보는 입력과 로그에서 배제합니다. 시간 로직에는 clock·sleep을 주입해 실제 대기 없이 재현하며, 공개 URL과 사실/해석 표기를 유지합니다.
