# 카카오페이증권 채택안-구현 정합성 검토

## 검토 결론

`docs/카카오페이증권/deep-research-report-01.md`의 최종 채택안은 반복적 MTS·해외주식 서비스 전산장애를 대상으로 한 `mts-stability-guard`다. `카카오페이증권/submission` 구현은 공개 장애 이력 분석, 모의 브로커 장애 재현, 방어 전후 비교, 실행별 상태·학습 기록, 회귀 테스트와 패키징 검증까지 포함하므로 채택안과 정합하다.

## 채택안 기준

- 반복적 MTS 장애와 해외주식 서비스 장애를 미국장 개장 피크, 외부 브로커 의존, 내부 시스템, 해외 지연으로 분류한다.
- 공개 이력과 모의 시스템만 사용해 장애를 재현하고 서킷브레이커, 재시도, 백프레셔, 헬스 게이트의 완화 효과를 검증한다.
- Codex manifest와 skill, 실행 코드, JSON·체크리스트 산출물, 방어 코드, 회귀 테스트를 제공한다.

초기 `docs/카카오페이증권/ulw-research-report-01.md.md`의 자동화 순위와 달리, 같은 문서의 재검증 및 deep-research 최종 판정은 문제 실재성과 Codex의 코딩·재검증 적합성을 근거로 MTS 안정성을 대표 제출작으로 채택했다. 보상, 공시 리스크, 자산관리 후보를 구현하지 않은 것은 최종 채택안과 충돌하지 않는다.

## I/O와 실패 계약

- 소스 checkout에서 기본 파이프라인 루트는 submission-level `input/`과 `output/`이다. 설치 레이아웃의 문서 명령은 `WORK_ROOT=$(pwd -P)`로 물리 경로를 구한 뒤 `--input-dir "$WORK_ROOT/input" --output-dir "$WORK_ROOT/output"`을 전달해 플러그인 설치 위치와 런타임 상태를 분리한다.
- 실행 입력과 config는 `input/<RUN_ID>/incidents.json`, `input/<RUN_ID>/stability-config.json`에 복사된다. 성공 산출물은 `output/runs/<RUN_ID>/analysis.json`, `chaos.json`, `scenario-checklist.md`, `state.json`과 `output/_learnings.md`다. `--skip-chaos`에서는 `chaos.json`을 만들지 않는다.
- 잘못된 RUN_ID, 중복 RUN_ID, 누락·malformed 입력은 실행 전 검증에서 거부해 해당 RUN_ID의 input/output 디렉터리와 성공 학습을 만들지 않는다. 디렉터리 예약과 입력 snapshot 이후 발생한 오류·중단·timeout만 `status=failed` state와 snapshot을 남기며 성공 학습을 append하지 않는다.
- RUN_ID는 `[A-Za-z0-9][A-Za-z0-9_.-]{0,127}` 형태의 단일 경로 성분이며, 선행 점과 `..`를 거부한다.
- 핵심 테스트 70개와 제출 계약 검사 6개는 method별 `input/test-runs/<TEST_ID>/input.json`과 `output/test-runs/<TEST_ID>/output.json`을 남긴다. 패키징 검사 5개는 생성물 없는 source tree를 읽기 전용으로 검사하므로, 전체 81개 중 이 계약의 예외다.

## 보안·배포 경계

- 실제 카카오페이증권 인프라, 고객 데이터, 계정, 라이브 브로커 API에 접근하지 않는다. 네트워크 호출과 제3자 런타임 패키지도 없다.
- 전용 secret/PII redaction 엔진은 없다. 공개·모의 데이터만 허용하고 계정, 토큰, 개인정보, 비공개 경로를 입력과 로그에서 배제한다.
- `input/`, `output/`, `.omo/`, `.codebase-memory/`, root `*-report.json`, 캐시와 bytecode는 실행·검증 산출물이지 배포 소스가 아니다. 원본 대화·QA 로그도 로컬에만 두며, 최종 제출 단계에서 공개 Git에 `logs/submission-evidence.jsonl` sanitized marker 하나만 추가한다. 패키징 검사가 이 경계를 확인한다.
- 실제 운영 안정성을 보장한다고 주장하지 않는다. 공개 장애 패턴을 모의 시스템에서 완화하고 회귀 검증할 수 있음을 입증한다.
- `.mcp.json`은 없다. 현재 skill과 로컬 실행 코드가 핵심 흐름을 완결하므로 선택적 MCP 통합을 구현한 것처럼 표현하지 않는다.

## 검증 명령과 기대값

- Python 3.9에서 핵심 5개 모듈을 명시 실행하면 70개 테스트가 통과해야 하며, 패키징 검사 5개와 제출 계약 검사 6개를 포함한 전체 discovery는 81개가 통과해야 한다.
- 생성물이 없는 새 clone/copy에서 `test_packaging_cleanliness.py`를 실행하면 5개 검사가 통과해야 한다. 임시 clone에 `output/runs/controlled/state.json`을 만들면 청결성 검사가 실패해야 한다.
- `analyze_incidents.py --help`, `chaos_runner.py --help`, `run_pipeline.py --help` 세 명령은 임의 작업 디렉터리에서 절대 플러그인 루트 경로로 실행 가능해야 한다.
- Codex plugin validator는 `src/.codex-plugin/plugin.json`과 skill frontmatter를 승인해야 한다. 임시 복사본에서 manifest JSON을 손상시키면 거부해야 한다.
- 분석 결과는 사건 5건과 권고 시나리오 4개를, chaos 결과는 3개 시나리오와 guarded 예외 누출 0을 유지해야 한다.
